# 门店商品数量口径对账分析

## 背景

本次分析围绕同一个门店 `ZZD2025` 的三套商品数量来源展开：

1. 三方平台前端页面统计接口 `/goods/store/tabSumV2`
2. 我方系统当前展示的“商品总数”
3. 三方平台导出的 Excel 明细 `C:\Users\ytsh01\Downloads\门店商品导出 (2).xlsx`

目标不是证明这些数字必须一致，而是确认它们各自代表什么口径，以及为什么会对不上。

---

## 一、三方平台前端统计接口口径

### 请求信息

- URL: `https://tc-supply-chain-portal.ele.me/goods/store/tabSumV2`
- Method: `POST`
- Request Body:

```json
{
  "storeCodes": "ZZD2025",
  "page": 1,
  "pageSize": 20,
  "tabType": "ALL",
  "unShelfType": 0,
  "excludeArchive": true
}
```

### 响应信息

```json
{
  "data": {
    "total": 14317,
    "saleTotal": 5240,
    "soldOutTotal": 9518,
    "downShelfTotal": 784,
    "unShelfTotal": {
      "total": 263,
      "initTotal": 0,
      "unShelfTotal": 0,
      "channelUnsaleTotal": 240
    },
    "hasErrorTotal": 4293
  },
  "status": true,
  "message": "OK"
}
```

### 结论

这组数据代表的是**三方前端页面自己的统计口径**。

其中：

- `14317 = data.total`：三方前端页面显示的门店商品总数
- `5240 = saleTotal`：可售/上架相关商品数
- `9518 = soldOutTotal`：售罄商品数
- `784 = downShelfTotal`：下架商品数
- `263 = unShelfTotal.total`：未上架相关商品数
- `4293 = hasErrorTotal`：异常商品数

注意：这些子项**不能简单相加**，因为它们大概率不是互斥分类。

## 二、我方系统当前展示口径

### 当前数值

- 我方系统当前“商品总数”：`14334`

### 当前推断

结合 Excel 反推结果，我方 `14334` 最像下面这个去重口径：

> **规范化商品名称 + 商品归属 + 商品类型**

其中“规范化商品名称”指：

- 去首尾空格
- 合并连续空格
- 统一常见全角/半角符号差异

这意味着：

- 同名商品，如果 `商品归属` 不同，则算不同商品
- 同名商品，如果 `商品类型` 不同，则算不同商品
- 如果只是名称空格或括号格式不同，则不应算不同商品

## 四、基于 Excel 的多种去重口径

### 1. 未做名称规范化

| 口径 | 数值 | 含义 |

| 商品名称 | 14319 | 同名算一件 |
| 商品名称 + 商品归属 | 14332 | 同名但总部/门店不同，算不同 |
| 商品名称 + 商品类型 | 14330 | 同名但普通/多规格/组合不同，算不同 |
| 商品名称 + 商品归属 + 商品类型 | 14336 | 同名但归属或类型不同，都算不同 |

| 口径 | 数值 | 含义 |

> **规范化商品名称 + 商品归属 + 商品类型**

## 五、关键差异如何理解

### 1. 为什么 Excel 16974 明显更大

因为它是**商品编码级明细**。

同一个展示商品，在 Excel 中可能因为以下字段不同而拆成多条：

- 商品归属不同（总部 / 门店）
- 商品类型不同（普通货品 / 多规格品 / 组合商品）
- 商品编码不同
- 售卖规格不同

### 2. 为什么 SPU 编码去重只有 2796

因为这批数据里的“商品总数”并不是按 SPU 在算。

`2796` 只能说明：

- 非空 SPU 编码只有 2796 个去重值
- 前端和我方系统的“商品总数”都不是 SPU 去重口径

### 3. 为什么 14319 也不能直接等于 14317

因为 `14319` 只是**原始商品名称去重**，而三方前端 `14317` 是页面统计口径。

两者只差 2，说明它们接近，但不能视为完全相同。最可能原因是：

- 页面还做了额外过滤
- `excludeArchive=true` 生效
- 某些页面不可见商品没有算进 total
- 某些商品在页面层被进一步合并

### 4. 为什么我方 14334 会高于三方前端 14317

差值：

- `14334 - 14317 = 17`

当前最可信的解释：

> 我方把一部分“同名但归属/类型不同”的商品算成了多个展示商品，而三方前端页面对其中一部分做了进一步的合并或过滤。

## 六、已经找到的可疑差异来源

### 1. 名称脏数据

仅通过名称规范化，可以把以下口径压缩：

- 原始商品名称去重：`14319`
- 规范化商品名称去重：`14313`

说明至少有 6 组名称只是因为：

- 多余空格
- 连续空格
- 全角/半角括号等格式差异

而被错误拆开。

### 2. 同名但多归属

统计结果：

- 存在 **16 个规范化商品名称**，同时拥有多个 `商品归属`

这类商品在 Excel / 我方统计中可能被拆成多条，但三方页面未必全部分开展示。

### 3. 同名但多类型

统计结果：

- 存在 **15 个规范化商品名称**，同时拥有多个 `商品类型`

典型冲突包括：

- 普通货品 vs 多规格品
- 普通货品 vs 组合商品

### 4. 同名但 `(商品归属, 商品类型)` 组合不同

统计结果：

- 存在 **21 个规范化商品名称**，拥有多个 `(商品归属, 商品类型)` 组合

这是当前最值得怀疑的一批候选：

> 这 21 组商品最可能解释为什么我方是 `14334`，而三方前端只有 `14317`。

## 七、当前最稳妥的口径定义

### 三方前端页面总数 14317

定义：

> 三方前端页面在 `tabType=ALL`、`excludeArchive=true` 等页面参数下显示的商品总数。

### 我方系统总数 14334

定义：

> 高可信接近“规范化商品名称 + 商品归属 + 商品类型”的商品总数。

### Excel 16974

定义：

> 商品编码级明细记录总数。

### SPU 非空去重 2796

定义：

> 非空 SPU 编码的唯一值数量。

---

## 八、关于“两个归属是否会导出成两条明细”

当前可以给出的结论是：

> **是，通常会。**

更准确地说：

- 如果同一个商品名同时存在两个 `商品归属`，导出中通常会出现两条或多条明细
- 如果再叠加 `商品类型`、`售卖规格`、`商品编码` 差异，则可能拆成不止两条

因此，Excel 明细本质上更像“展示商品的明细展开结果”，而不是页面展示层的聚合商品数。

---

## 九、当前结论总结

1. 三方前端页面 `14317` 是页面自己的统计口径，不等于 Excel 明细行数，也不等于 SPU 数。
2. Excel `16974` 是商品编码级别的明细数，一行基本对应一条商品编码记录。
3. 我方系统 `14334` 最像“规范化商品名称 + 商品归属 + 商品类型”的统计口径。
4. 三方前端 `14317` 比我方 `14334` 少 17，最可能来源于页面层对部分重复展示商品的进一步合并或过滤。
5. 当前最可疑的候选是那 **21 组同名但 `(商品归属, 商品类型)` 组合不同** 的商品。
