# 零售人-货-场标签体系完整结构化字典

## 总体层级说明

- **L1 领域**：最高层业务视角切分
- **L2 维度类**：领域内的属性大类
- **L3 原子维度**：不可再分的独立分析轴
- **L4 标签值**：具体可枚举的取值

打标方式说明：
- **规则**：基于明确规则自动打标（如类目映射、阈值分段）
- **算法**：基于机器学习模型打标
- **人工**：人工标注或配置
- **继承**：从主数据系统直接同步

## 零售人-货-场标签体系：设计思想与架构白皮书

### 一、这是什么？

这是一套**覆盖零售核心业务对象（商品、门店/渠道、消费者）的原子化、正交化、可组合的标签体系架构**。

它不是：
- 一份简单的“字段清单”或“数据字典”
- 一堆随意堆砌的业务形容词
- 一个依赖人工记忆和维护的经验库

它是：
- **一套业务语义与数据存储解耦的标准语言**：让业务人员能用“户外露营”、“高价值活跃”、“坪效偏低”这种自然语言驱动数据，而无需关心底层表结构。
- **一个可生长、可治理的有机系统**：有明确的扩展原则、生命周期管理机制和人机协同生产方式。
- **一条连接“人货场”的数字化总线**：通过统一的ID和标签值，实现三者的任意交叉分析和策略联动。

**整体规模速览**：

| 标签层 | 领域数 | 维度类数 | 原子维数 | 典型标签值数 |
| :--- | :--- | :--- | :--- | :--- |
| 货（商品） | 4 | 8 | 29 | 200+ |
| 场（门店/渠道） | 4 | 15 | 45 | 300+ |
| 人（消费者） | 5 | 19 | 60 | 400+ |

### 二、为什么这样设计？——核心架构原则

#### 原则一：原子化与组合化并存（乐高原则）

**原子化**：每个标签维度是不可再分的独立分析轴。例如，“使用空间：户外露营”绝不包含“烹饪”或“便携”的语义，后者属于“核心动作”和“功能痛点”维度。

**组合化**：复杂业务语义通过**虚拟标签（聚合标签）** 来封装，例如“露营核心装备 = 使用空间:户外露营 + 核心动作:烹饪 + 功能痛点:便携轻量”。

> **为何如此？**  
> - **底层灵活**：允许业务人员用任意逻辑组合29个原子维，应对不断变化的运营需求。  
> - **上层简单**：常用组合一键圈选，降低使用门槛。

#### 原则二：视角正交化（独立切面原则）

每个L1领域代表一个独立的业务分析视角，领域之间不重叠、不混淆。

- **货的四个视角**：内部经营（运营管理）、用户使用（消费场景）、用户购买（消费决策）、商品关联（商品关系）。
- **人的五个视角**：他是谁（基础属性）、他做什么（行为轨迹）、他怎么买（消费特征）、他有多大价值（价值评估）、他可能喜欢什么（偏好预测）。

> **为何如此？**  
> - 避免“健康成分”和“功能痛点”被混为一谈，前者属于消费场景视角，不应独立成领域。  
> - 保证体系在增长时不会因概念交叉而失序。

#### 原则三：动态状态与静态属性分离（时效性分治原则）

- **静态属性类**（如商品角色、品牌层级、开业年限）：由人工或规则一次性/低频维护。
- **高动态状态类**（如库存状态、活跃度、实时流量状态）：由规则或算法按小时/天级自动刷新。

> **为何如此？**  
> - 避免运营人员每天手动修改上万商品的“库存状态”标签，那是机器的职责。  
> - 避免晨会看板无法反映真实缺货和投诉风险。

#### 原则四：业务语义与数据存储解耦（翻译层原则）

标签命名面向业务人员（“银发族”而非“age>55”），底层存储是联合主键的标签关联表。虚拟标签不存储数据，只存储**逻辑表达式**。

> **为何如此？**  
> - 品类运营可以直接点击“露营核心装备”圈选，无需写SQL。  
> - 系统层面不产生数据冗余，标签变更不会引发大量数据回刷。

### 三、这样设计会如何？不这样设计会如何？——价值对比

| 对比维度 | **采用本体系（原子化+正交化+虚拟层）** | **不采用本体系（传统字段清单/随意打标）** |
| :--- | :--- | :--- |
| **圈选灵活性** | 运营可自由组合29个维度，满足任意细分场景需求 | 只能使用预设的有限几个大类标签，新需求必须找IT改代码 |
| **跨部门协作** | 全公司对“爆款”“高价值”“新店”有唯一定义 | 市场部、采购部、商品部对“爆款”各自定义，会议撕扯 |
| **体系生长性** | 新增原子维度需通过“四问检验法”，架构不乱 | 各部门随意建标签，一年后标签库有5000个无人能懂的标签 |
| **运营效率** | 虚拟标签一键圈选，活动选品从天级降到分钟级 | 每次活动前手动翻Excel、凭记忆选品，耗时且易漏 |
| **数据质量** | 标签覆盖率、使用率、效果归因均有监控看板 | 大量僵尸标签存在，数据团队不知道哪些标签在生产中真正有用 |
| **人财物投入** | 初期需投入1-2个月架构设计，但长期维护成本极低 | 初期看似“快速上线”，后期持续陷入打标混乱和高昂治理成本 |

### 五、如何操作与使用？

#### 5.1 标签生产分工（人机协同流水线）

| 生产角色 | 负责内容 | 典型工具/手段 |
| :--- | :--- | :--- |
| **系统自动继承** | 从主数据系统同步基础字段（品类、品牌等） | 数据同步任务（DataX/Airbyte） |
| **规则引擎** | 基于类目、标题、属性值、统计阈值分档的自动化打标 | 规则配置平台（类Flink SQL / Groovy脚本） |
| **算法模型** | 挖掘评价情感、关联关系、流失概率、LTV预测 | 机器学习平台 + 调度系统 |
| **人工标注** | 复核头部品、配置商品角色/主推状态、标记特殊场景 | 标签管理后台（类似商品编辑页） |

#### 5.2 运营人员使用三步法

1. **选对象**：在运营后台选择“圈人”“圈品”或“圈店”。
2. **定条件**：
   - **方式A（组合圈选）**：展开标签树，勾选原子维度标签值，支持AND/OR组合。
   - **方式B（一键圈选）**：点击“虚拟标签”卡片，系统自动解析为原子条件。
3. **执行动作**：预览人群规模后，一键生成人群包，用于活动投放、Push推送或商品上下架规则。

#### 5.3 管理者视角看板

| 层级 | 看板内容 |
| :--- | :--- |
| **总部运营管理** | 全量商品/门店/会员的标签覆盖统计、虚拟标签使用排行、各领域标签分布健康度 |
| **数据分析师** | 原子维度的覆盖率趋势、标签值与业务指标的关联分析（如“高曝光低转化”品的后续干预效果） |
| **数据治理** | 僵尸标签预警清单、标签值冲突检测、各来源打标一致性监控 |

### 六、如何迭代更新？

#### 6.1 日常迭代：L4标签值的新增与下线

- **新增流程**：业务方在后台提交申请 → 系统自动检测是否与现有标签值同义 → 数据产品审核唯一性和定义清晰度 → 配置上线。
- **下线流程**：系统每周扫描覆盖SKU数和使用频次 → 连续90天双零则进入“待下线”观察 → 业务方确认无用后标记“废弃”。

#### 6.2 结构演进：L3原子维度的审慎新增

必须通过 **“四问检验法”** ，由数据架构师和业务方共同评审：

| 问题 | 不通过时的替代方案 |
| :--- | :--- |
| 能否用现有原子维度组合表达？ | 能 → 创建虚拟标签，不新增维度 |
| 该维度是否还能继续拆分？ | 能 → 拆分为两个或多个原子维度 |
| 与现有维度是否有语义重叠？ | 有 → 重新定义边界或合并 |
| 能否以可接受成本实现≥60%覆盖率？ | 否 → 暂缓，列入技术Roadmap |

#### 6.3 架构升级：L2/L1层级的极端情况

仅当企业发生战略级业务变化（如从零售切入餐饮服务）时，才启动L1领域或L2维度类的增删。需经过数据委员会最高层级评审，并先在虚拟标签层试跑3个月以上验证价值。

### 七、核心原则速查卡

| 编号 | 原则名称 | 一句话解释 |
| :--- | :--- | :--- |
| P1 | **原子化优先** | 能拆成两个独立的轴，就不合并成一个模糊的标签。 |
| P2 | **组合优于新建** | 能用现有原子组合表达的复杂概念，一律建虚拟标签。 |
| P3 | **正交化视角** | L1领域是互不重叠的分析切面，不能有“杂物间”领域。 |
| P4 | **动静态分离** | 高频变动的状态类标签，必须机器自动更新，不依赖人工。 |
| P5 | **业务语义解耦** | 标签是给业务人员看的，底层是技术存储，中间有翻译层。 |
| P6 | **生产可行性定生死** | 打不出来的标签，再好的概念也暂缓落地。 |
| P7 | **生命周期闭环** | 标签不仅要建，更要监控使用率、主动下线僵尸标签。 |

### 八、总结：这套体系的本质

**以原子化为骨架，以正交化为神经，以虚拟层为皮肤，以扩展机制为生长边界**——这是一套能将零售企业的人、货、场数据资产，从散乱的“信息噪音”转化为有序、可运营、可度量的“决策语料”的基础设施。

它解决的不仅仅是一个“怎么打标签”的技术问题，而是一个 **“企业如何形成统一的数据语言，并让每个业务角色都能轻松使用这种语言驱动增长”** 的组织级命题。
