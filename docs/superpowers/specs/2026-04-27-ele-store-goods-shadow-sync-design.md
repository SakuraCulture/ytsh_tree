# 饿了么门店商品影子同步设计

## 背景

当前饿了么门店商品同步链路已经可以通过 `saas.goods.store.query.batch` 分页拉取门店商品，并在同步时用三方返回的 `sku_code` 匹配本地 `SkuTableDO.productSkuCode`。匹配成功后写入 `store_product_table`，匹配失败则进入治理池。

现在确定存在新的外部约束：三方 API 未直接提供商品主档全部信息的全量分页拉取接口，导致本地主档同步能力受阻。继续要求门店商品同步强依赖本地主档，会导致大量真实存在的三方门店商品无法进入系统，只能沉淀为 `SKU_NOT_FOUND` 治理记录。

本次设计目标是调整门店商品同步链路，让三方门店商品即使缺少本地主档也可以进入正式门店商品列表，并以“主档缺失 / 信息不完整”状态参与后续运营。

## 目标

1. 门店商品同步不再因为本地 SKU 主档缺失而整体失败。
2. 未匹配本地主档的三方门店商品可以进入正式门店商品列表展示。
3. 未匹配数据必须明确标记为不完整，避免误认为已经接入完整商品体系。
4. 后续主档补齐后，系统可以自动或人工归并影子数据到正式门店商品。
5. 保护现有 `store_product_table` 的正式业务语义，避免直接污染商品主档和门店商品主表。
6. 数据一致性、列表性能和归并可重试性必须有明确边界。

## 不建设范围

1. 不建设三方商品主档全量同步能力。
2. 不自动创建占位 SPU / SKU 主档。
3. 不把 `store_product_table.product_sku_id` 改成可空来承载未匹配数据。
4. 不在列表查询时实时执行复杂主档匹配。
5. 不在本阶段建设完整规则引擎或算法归因能力。

## 推荐方案

采用“平台门店品影子表 + 正式列表聚合 + 异步归并”的方案。

新增一张平台门店品快照表，用来承接三方接口返回的门店商品事实。同步链路先落快照，再尝试匹配本地 SKU：

- 匹配成功：继续写入或更新 `store_product_table`，并把快照标记为已归并。
- 未匹配：保留在影子表，标记为主档缺失，但仍参与门店商品列表展示。
- 多命中或字段冲突：保留在影子表，标记为冲突，进入治理状态。

正式门店商品列表由正式表和影子表聚合而成：正式表记录优先，影子表只补充未归并、未被正式记录覆盖的三方门店品。

## 方案取舍

### 方案一：扩展正式表

直接改造 `store_product_table`，允许 `product_sku_id` 为空，并增加三方 `sku_code`、`spu_code`、完整性状态等字段。

优点：

- 查询链路短。
- 前端改造相对直接。

缺点：

- 破坏现有正式门店商品模型。
- 现有服务、仓储、库存、商品联动逻辑普遍假设存在本地 SKU。
- 后续主档补齐后的归并和清理边界不清晰。

### 方案二：自动创建占位 SKU

同步时发现本地 SKU 缺失，则基于三方门店商品信息创建占位 SPU / SKU，再按现有门店商品链路写入正式表。

优点：

- 列表和操作复用最多。
- 性能接近当前单表模型。

缺点：

- 污染商品主档。
- 三方门店商品信息不完整，无法保证占位主档质量。
- 后续真实主档到达时需要处理占位主档归并，风险高。

### 方案三：影子表聚合

新增平台门店品快照表，正式表保持语义不变，列表层聚合正式数据与未归并影子数据。

优点：

- 不污染商品主档和正式门店商品表。
- 能承接主档缺失、三方脏数据、乱序到达等异常事实。
- 后续主档同步恢复后，可以通过状态机和归并任务平滑收口。

缺点：

- 需要新增表、状态机、聚合查询和归并任务。
- 列表和操作层需要区分正式记录与影子记录。

本次选择方案三。

## 数据模型

### 平台门店品快照表

建议新增 `ele_store_goods_shadow`，用于保存三方门店商品快照。

核心字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `platform_id` | 平台 ID，饿了么固定为 1 |
| `merchant_code` | 商家编码 |
| `erp_store_code` | ERP 门店编码 |
| `platform_store_id` | 平台门店编码 |
| `store_id` | 本地门店 ID，可空但尽量同步解析 |
| `spu_code` | 三方 SPU 编码 |
| `sku_code` | 三方 SKU 编码 |
| `sub_sku_code` | 店内 SKU 编码 |
| `title` | 商品名称快照 |
| `main_pic` | 主图快照 |
| `sub_pics` | 副图快照，JSON 或文本 |
| `front_category` | 前台类目快照，JSON 或文本 |
| `brand_name` | 品牌名称快照 |
| `specification` | 规格名称快照 |
| `sale_price` | 门店销售价 |
| `pos_status` | 平台上下架状态映射值 |
| `is_active` | 是否启用 |
| `raw_payload` | 原始门店商品或 SKU 报文 |
| `match_status` | 匹配状态 |
| `matched_product_sku_id` | 已匹配的本地 SKU ID |
| `merged_store_product_id` | 已归并的正式门店商品 ID |
| `last_sync_time` | 最近同步时间 |
| `matched_time` | 匹配时间 |
| `merged_time` | 归并时间 |
| `conflict_reason` | 冲突原因 |
| `creator/create_time/updater/update_time/deleted/tenant_id` | Yudao 标准字段 |

### 唯一键

同一平台、同一商家、同一门店、同一 SKU 只能有一条当前快照：

`platform_id + merchant_code + erp_store_code + sku_code + deleted`

如果后续确认 `sub_sku_code` 才是门店内唯一编码，可以追加到唯一键或作为冲突判断字段。但第一版不建议依赖 `sub_sku_code` 做主唯一键，因为现有同步链路使用 `sku_code` 匹配本地 SKU。

### 状态机

`match_status` 建议使用以下状态：

| 状态 | 说明 |
| --- | --- |
| `UNMATCHED` | 未匹配本地主档，参与正式列表展示并标记信息不完整 |
| `MATCHED` | 已唯一匹配本地 SKU，待归并或可归并 |
| `MERGED` | 已写入正式 `store_product_table`，默认不再作为影子记录展示 |
| `CONFLICT` | 多命中、编码冲突或字段冲突，需要人工治理 |
| `IGNORED` | 人工确认不处理，默认不进入正式列表 |

## 同步数据流

### 当前全量 / 分页同步

1. 调用 `saas.goods.store.query.batch` 拉取门店商品分页。
2. 遍历返回的 `ChannelGoodsSyncReq + ChannelSkuSyncReq`。
3. 以 `platform_id + merchant_code + erp_store_code + sku_code` upsert 影子表。
4. 尝试用 `sku_code` 匹配本地 `SkuTableDO.productSkuCode`。
5. 如果唯一匹配：
   - 写入或更新 `store_product_table`；
   - 影子表更新为 `MERGED`；
   - 记录 `matched_product_sku_id`、`merged_store_product_id`、`matched_time`、`merged_time`。
6. 如果未匹配：
   - 影子表状态为 `UNMATCHED`；
   - 同步结果计为影子成功，不再把整条同步视为失败；
   - 同步写入或更新治理提示记录，治理池作为处理入口，影子表作为事实承载。
7. 如果多命中或冲突：
   - 影子表状态为 `CONFLICT`；
   - 记录冲突原因；
   - 进入治理。

### 后续主档补齐归并

新增异步归并任务或手动归并入口：

1. 扫描 `UNMATCHED`、`CONFLICT` 中可重试的数据。
2. 按 `sku_code` 查找本地 SKU。
3. 唯一命中时在事务内：
   - 锁定或重新读取影子记录，确认状态未被其他任务处理；
   - upsert `store_product_table`；
   - 更新影子记录为 `MERGED`。
4. 多命中或字段冲突时保留 `CONFLICT`。
5. 找不到时保持 `UNMATCHED`。

## 列表聚合设计

门店商品列表需要返回统一行模型，并区分来源：

| 字段 | 说明 |
| --- | --- |
| `rowSource` | `FORMAL` 或 `SHADOW` |
| `storeProductId` | 正式门店商品 ID，影子记录为空 |
| `shadowId` | 影子记录 ID，正式记录为空 |
| `storeId` | 本地门店 ID |
| `productSkuId` | 本地 SKU ID，影子未匹配时为空 |
| `skuCode` | SKU 编码，正式记录来自主档，影子记录来自三方 |
| `productName` | 商品名，正式记录来自主档，影子记录来自快照 |
| `price` | 门店价格 |
| `posStatus` | 门店上下架状态 |
| `completenessStatus` | `COMPLETE` 或 `MASTER_MISSING` |
| `matchStatus` | 影子匹配状态，正式记录可为空或固定 `MERGED` |

聚合规则：

1. 正式表记录优先。
2. `MERGED` 和 `IGNORED` 影子记录默认不参与主列表。
3. 如果同门店同 `sku_code` 已存在正式记录，未归并影子记录不重复展示，只保留治理入口可见。
4. 默认列表展示 `FORMAL` + `UNMATCHED` + `CONFLICT`。
5. 筛选条件需要支持完整状态、匹配状态、门店、SKU 编码、商品名。

## 操作语义

用户选择了“未匹配主档的门店商品允许全部现有操作”。因此前端入口尽量保持一致，但后端必须按记录来源分流：

- `FORMAL`：沿用现有正式门店商品操作。
- `SHADOW`：操作保存到影子表对应字段或影子操作记录中，不要求立即存在本地 SKU。

第一版建议至少支持以下影子字段更新：

1. 价格。
2. 上下架状态。
3. 是否启用。
4. 备注或治理状态。

涉及库存、标签、订单归因、仓储联动等强依赖本地 SKU 的功能，第一版不直接复用正式表写接口。接口层必须识别 `rowSource=SHADOW`，并把可表达的门店侧字段保存到影子表；无法脱离本地 SKU 安全执行的操作返回明确错误。归并成功后，再由归并任务把影子侧保存的价格、上下架和启用状态同步到正式门店商品。

## 一致性设计

### 事实源边界

- `store_product_table` 是正式业务事实。
- `ele_store_goods_shadow` 是平台门店品事实快照。
- 影子表不代表商品主档完整存在。
- 列表聚合只是展示和运营入口，不改变两类事实的边界。

### 去重规则

同一门店同一 `sku_code` 同时存在正式记录和影子记录时：

1. 默认展示正式记录。
2. 影子记录保留在治理视图或详情中。
3. 如果影子记录状态不是 `MERGED`，需要提示存在待处理平台快照，避免数据被完全隐藏。

### 归并事务

归并必须在单个事务内完成：

1. 确认影子记录仍处于可归并状态。
2. 确认 SKU 唯一匹配。
3. upsert 正式门店商品。
4. 更新影子状态为 `MERGED`。

事务失败时，影子记录不得提前标记为 `MERGED`。

### 乱序更新

三方接口如果没有可靠更新时间，则使用本地 `last_sync_time` 表示最后接收时间。相同唯一键的重复同步采用最后写入策略，并保留最新原始报文。

如果后续三方补充版本号或更新时间，再升级为按外部版本覆盖，避免旧消息覆盖新消息。

## 性能设计

### 查询策略

第一版不建议做大范围数据库 `UNION`。服务层可以按筛选条件分别查询正式表和未归并影子表，再进行统一行模型组装。

为了避免分页不准确，推荐第一版分页策略为：

1. 有门店、SKU、商品名、完整状态等筛选条件时，分别查询两侧候选数据。
2. 服务层按统一排序字段合并、去重、截断分页。
3. 对默认无筛选的大列表，限制必须带门店或状态筛选，或使用默认门店范围，避免扫描过大。

如果后续数据规模明显增大，再引入数据库视图、物化索引表或搜索索引。

### 索引建议

影子表建议建立以下索引：

1. 唯一索引：`platform_id + merchant_code + erp_store_code + sku_code + deleted`。
2. 查询索引：`store_id + match_status + deleted`。
3. 查询索引：`platform_id + erp_store_code + match_status + deleted`。
4. 匹配索引：`sku_code + match_status + deleted`。
5. 任务索引：`match_status + update_time + deleted`。

商品名搜索如果成为高频场景，再单独评估前缀索引或搜索能力，不在第一版过度建设。

### 归档策略

`MERGED` 影子记录默认不参与列表聚合。后续可以按时间归档或清理，例如归并超过 90 天的数据迁移到历史表或仅保留同步日志。

第一版先不自动删除，避免影响排障。

## 治理设计

治理池不再只是 `SKU_NOT_FOUND` 的唯一落点，而是影子状态的辅助工作台。

建议治理入口支持：

1. 查看 `UNMATCHED` 门店品。
2. 查看 `CONFLICT` 门店品。
3. 手动选择本地 SKU 后归并。
4. 人工忽略，状态改为 `IGNORED`。
5. 查看原始三方报文和最近同步时间。

## 前端展示

门店商品列表新增以下展示能力：

1. 信息完整性标签：完整 / 主档缺失。
2. 数据来源标签：正式 / 平台快照。
3. 匹配状态筛选：未匹配、冲突、已归并、已忽略。
4. 影子记录详情：展示三方返回的名称、图片、类目、规格、价格、原始报文。

对影子记录的操作入口保持和正式记录一致，但需要在详情或列表上明确提示“该记录来源于平台快照，尚未匹配本地主档”。

## 测试策略

### 后端单元测试

1. 三方分页同步返回本地已存在 SKU 时，写入正式表并把影子记录标记为 `MERGED`。
2. 三方分页同步返回本地不存在 SKU 时，写入影子表 `UNMATCHED`，同步页结果不计为失败。
3. 同一门店同一 `sku_code` 重复同步时更新同一条影子记录。
4. 归并任务唯一命中 SKU 时，在事务内写正式表并更新影子状态。
5. 归并任务多命中或冲突时，影子状态变为 `CONFLICT`。
6. 列表聚合时正式记录优先，影子记录不重复展示。
7. `MERGED` 和 `IGNORED` 默认不进入主列表。

### 前端验证

1. 门店商品列表可以展示正式记录和主档缺失记录。
2. 主档缺失记录有明显标识。
3. 完整性和匹配状态筛选可用。
4. 影子记录详情能展示平台快照信息。
5. 对影子记录执行现有操作时，页面能得到明确结果或明确错误。

### 集成验证

1. 使用测试模式全量同步某门店，确认未匹配 SKU 不再只进入失败日志。
2. 后续补齐 SKU 后运行归并任务，确认影子记录从 `UNMATCHED` 变为 `MERGED`。
3. 列表中归并后的记录只显示正式记录，不重复显示影子记录。

## 结论

本次调整采用影子表聚合方案。它比直接扩展正式表或创建占位 SKU 更复杂，但能保护现有商品主档和正式门店商品表的语义，适合当前“三方门店品可分页、主档全量受阻”的外部约束。

影子表在当前阶段承担展示和运营入口；当主档链路恢复后，它会降级为同步缓冲区和异常治理区。通过明确状态机、唯一键、正式表优先规则和事务归并机制，可以控制数据一致性风险，并为后续门店商品标签、治理和运营能力保留清晰扩展边界。
