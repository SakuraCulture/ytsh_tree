# 商品标签体系全量挂载设计

## 背景

当前仓库已经具备商品标签体系的一期元数据中心能力：

- 标签维度树：`tag_dimension`
- 标签值：`tag_value`
- 虚拟标签：`tag_virtual`
- 对象关系：`tag_object_relation`
- 商品主档 SPU 标签管理：`/business/product-spu-tag`

但现状仍有三个明显缺口：

1. `PRODUCT` 域下实际只支持 `SPU` 对象，关系层没有正式放开到其它商品对象。
2. 门店商品 `store_product_table` 已有完整对象和后台页面，但没有接入标签挂载链路。
3. 商品主档和门店商品之间还没有形成统一的“可打、可看、可改、可回显、可筛选”的运营闭环。

本次作业目标是把商品维度标签体系完整挂载到两类商品对象：

- 商品主档（SPU）
- 门店商品（STORE_PRODUCT）

同时复用现有商品标签字典，不新开第二套门店商品标签池。

## 目标

1. 保留现有 `PRODUCT` 域标签字典，支持同时挂载到 `SPU` 和 `STORE_PRODUCT`。
2. 商品主档和门店商品都支持手工标签的查询、编辑、保存、回显。
3. 商品主档和门店商品列表都支持标签展示与按标签筛选。
4. 商品主档和门店商品都支持批量挂标入口，满足一期运营批量操作场景。
5. 复用统一的对象关系表 `tag_object_relation`，不为门店商品新开独立关系表。
6. 为后续规则来源、批量挂标扩展、SKU 扩展预留同一套对象类型扩展路径。

## 非目标

1. 不在本次改动中新增独立的门店商品标签字典。
2. 不在本次改动中实现 SPU 标签向门店商品的自动继承。
3. 不在本次改动中实现门店商品标签反推 SPU。
4. 不在本次改动中实现规则引擎、算法标签生产、自动打标任务。
5. 不在本次改动中把标签直接落到 SKU 层。
6. 不在本次改动中实现 Excel 导入式批量挂标；一期只要求后台批量挂标入口闭环，不要求文件导入批量打标签。

## 现状摘要

### 已有能力

#### 1. 标签元数据中心
业务模块已经具备以下实体和接口：

- `tag_dimension`：维护 L1/L2/L3 标签维度树
- `tag_value`：维护 L4 标签值与打标方式、数据源、更新频率、逻辑说明
- `tag_virtual`：维护虚拟标签定义和表达式 JSON
- `tag_object_relation`：维护对象与标签值关系

#### 2. 商品主档 SPU 标签挂载
已有 SPU 标签接口：

- 查询 SPU 标签列表
- 批量查询 SPU 简要标签列表
- 保存 SPU 手工标签

#### 3. 门店商品对象和后台页面
门店商品已有独立业务对象、列表页、详情/编辑页、分页接口和导入导出能力：

- 表：`store_product_table`
- 后端控制器：`/store-product`
- 前端页面：`ytsh-ui-vue3/src/views/business/store-product/index.vue`

### 当前缺口

#### 1. 对象关系层只支持 SPU
当前 `TagConstants` 和关系服务虽然保留了扩展接口，但实际上只允许 `OBJECT_TYPE_SPU`。

#### 2. 门店商品没有任何标签挂载链路
门店商品页面没有标签入口，后端也没有对称的标签控制器和服务。

#### 3. 列表页没有形成运营闭环
SPU 和门店商品都缺少稳定的一期运营入口组合：

- 标签回显
- 标签列展示
- 按标签筛选
- 批量简要标签查询
- 批量挂标入口

## 总体方案

采用“单域单池、多对象类型、统一关系表”的方案。

### 核心决策

#### 1. 保留单一商品域
继续使用：

- `domainType = PRODUCT`

不新增 `STORE_PRODUCT` 独立域，不拆门店商品专用标签池。

#### 2. 扩展商品域对象类型
把 `PRODUCT` 域下的对象类型从当前的单一 `SPU` 扩展为：

- `SPU`
- `STORE_PRODUCT`

#### 3. 继续复用统一关系表
继续使用 `tag_object_relation` 保存挂载关系：

- `domain_type = PRODUCT`
- `object_type = SPU | STORE_PRODUCT`
- `object_id = productSpuId | storeProductId`

#### 4. 两类对象独立挂标，不做自动继承
本次一期明确：

- SPU 标签是商品主档全局视角标签
- 门店商品标签是门店经营上下文视角标签
- 两类对象共享标签池，但各自独立保存和查询标签关系
- 不做自动继承、自动聚合、自动回写

这样可以避免现在就引入标签优先级、覆盖规则、同步时机等复杂问题。

## 对象边界与语义

### SPU 标签语义
SPU 标签表示商品主档本身具备的属性和运营定义，例如：

- 爆品
- 露营场景
- 高价值商品
- 健康控糖

这类标签面向全局商品定义，不依赖具体门店。

### 门店商品标签语义
门店商品标签表示某个商品在某个门店经营语境下的标签，例如：

- 门店重点陈列
- 门店试销
- 社区店适销
- 门店清仓

这类标签允许与 SPU 标签并存，但不互相覆盖。

## 数据模型设计

### 1. 常量扩展
扩展 `TagConstants`：

- 新增 `OBJECT_TYPE_STORE_PRODUCT = "STORE_PRODUCT"`
- `OBJECT_TYPES` 从仅包含 `SPU` 改为包含 `SPU`、`STORE_PRODUCT`

`DOMAIN_TYPES` 保持不变，仍以 `PRODUCT` 作为商品域。

### 2. 对象关系表
`tag_object_relation` 表结构保持不变，不新增字段。

当前表结构已经满足：

- 域类型
- 对象类型
- 对象 ID
- 标签值 ID
- 来源类型
- 来源引用
- 生效/失效时间

本次只扩展其业务语义：允许 `PRODUCT` 域下挂到两类对象。

### 3. 标签字典
`tag_dimension`、`tag_value`、`tag_virtual` 表结构保持不变。

本次明确规则：

- 门店商品与 SPU 共用同一套 `PRODUCT` 域标签值
- 不新增门店商品专属标签表
- 不新增对象类型字段到标签值表

后续如需做“部分标签只允许门店商品用”的限制，可以在服务层和页面层增加对象类型过滤规则，不要求本次修改表结构。

## 后端设计

## 总体分层

分为三层：

1. 标签元数据层：继续维护标签维度、标签值、虚拟标签
2. 对象关系层：统一处理 `SPU` / `STORE_PRODUCT` 的标签挂载
3. 业务对象入口层：分别提供 SPU 和门店商品的标签接口

## 关系服务改造

### TagObjectRelationServiceImpl
把现有关系服务改造成真正按 `objectType` 工作：

#### 校验规则
- `domainType` 仍要求合法枚举
- `PRODUCT` 域下允许 `SPU`、`STORE_PRODUCT`
- 其它域不在本次扩展范围内

#### 标签值校验
保存标签前继续校验：

1. 标签值存在
2. 标签值状态为启用
3. 标签值所属维度存在
4. 维度属于 `PRODUCT` 域
5. 维度层级必须是 L3
6. 维度状态为启用

#### 保存语义
手工保存仍采用“本对象、本来源覆盖式保存”：

- 本次提交的手工标签集合，代表该对象当前完整手工标签集合
- 提交中不存在的旧手工标签关系会被禁用
- 提交中的标签关系会插入或恢复启用

这套语义对 `SPU` 和 `STORE_PRODUCT` 保持一致。

## 标签值服务改造

### TagValueServiceImpl
当前 `getSelectableTagValuesForObject(objectType)` 虽然收 `objectType` 参数，但实际上只支持 `SPU`。

本次改为：

- 接受 `SPU` 和 `STORE_PRODUCT`
- 两者都返回 `PRODUCT` 域下启用的 L3 标签值

因为本次决策是“直接共用商品标签池”，所以两种对象类型默认返回相同的标签候选集。

## 数据流设计

## 保存标签

### SPU
1. 前端请求商品域标签池
2. 用户勾选标签并提交
3. 后端校验 `productSpuId`
4. 后端校验标签值合法性
5. 统一关系服务以 `PRODUCT + SPU` 保存手工关系

### STORE_PRODUCT
1. 前端请求商品域标签池
2. 用户勾选标签并提交
3. 后端校验 `storeProductId`
4. 后端校验标签值合法性
5. 统一关系服务以 `PRODUCT + STORE_PRODUCT` 保存手工关系

### 批量挂标
1. 前端在列表页选择多个对象
2. 打开批量标签弹窗并加载商品域标签池
3. 用户提交统一标签集合
4. 后端逐个校验对象存在性与对象类型合法性
5. 统一关系服务对每个对象执行手工标签全集覆盖保存
6. 返回成功数、失败数和失败明细

批量挂标只支持正式对象，不支持门店商品影子行。

## 列表回显

1. 主列表先查对象主数据
2. 再批量查询这些对象的简要标签
3. 前端按对象 ID 挂回到对应行

这样可以避免一行一查导致的 N+1 问题。

## 校验规则

本次要求以下规则明确写死：

1. `PRODUCT` 域允许对象类型：`SPU`、`STORE_PRODUCT`
2. `SPU` 保存前必须校验商品主档存在
3. `STORE_PRODUCT` 保存前必须校验门店商品存在
4. 标签值必须属于 `PRODUCT` 域
5. 标签值必须挂在 L3 维度下
6. 标签值和维度都必须为启用状态
7. 手工保存采用覆盖式更新当前手工标签全集
8. 门店商品影子行不允许进行标签挂载

## 风险与控制

### 风险 1：同一套商品标签池被门店商品过度滥用
问题：全局商品语义标签可能被错误地挂到门店经营对象上。

控制：

- 本次先不拆标签池
- 保留 `objectType` 参数和服务入口
- 后续若需要做对象级可选标签限制，可在不改表结构的前提下追加服务层过滤

### 风险 2：列表筛选性能
问题：标签筛选如果直接做成复杂分页 SQL，可能拖慢列表。

控制：

- 一期优先做清晰、可维护的标签关系反查
- 只在必要时为筛选追加专用 mapper
- 不把两类对象的筛选逻辑揉成一条复杂 SQL

### 风险 3：后续 SKU 层扩展
问题：长期看某些规格维度可能需要落 SKU。

控制：

- 本次先把对象类型模型做成可扩展
- 后续如需加 `SKU`，只是在同一模型上新增第三类对象类型
- 不推翻本次 SPU / STORE_PRODUCT 的统一关系设计

### 风险 4：门店商品影子对象的混入
问题：影子对象不是正式经营对象，如果放进标签体系会污染运营语义。

控制：

- 一期明确影子对象不参与标签挂载
- 页面和接口都只针对正式门店商品开放标签操作

## 验收标准

满足以下条件即可判定本次“商品维度标签体系全量挂载一期”完成：

1. 商品主档可以查看、编辑、保存、回显手工标签
2. 商品主档支持批量挂标
3. 门店商品可以查看、编辑、保存、回显手工标签
4. 门店商品支持批量挂标
5. 两类对象共用同一套商品标签字典
6. 两类对象列表都能展示标签
7. 两类对象列表都能按标签筛选
8. 后端关系模型正式支持 `SPU + STORE_PRODUCT`
9. 关键成功路径与核心校验失败路径都有测试覆盖
10. 影子门店商品不进入本次标签挂载范围

## 实施建议顺序

建议按以下顺序落地：

1. 扩展常量与关系服务对象类型支持
2. 新增门店商品标签后端接口与服务
3. 补充 SPU / STORE_PRODUCT 标签批量简要查询与筛选能力
4. 补前端商品主档列表标签展示与筛选
5. 补前端门店商品标签入口、展示与筛选
6. 完成后端测试与前端验证

## 结论

本方案以“单域单池、多对象类型、统一关系表”为核心，把商品标签体系正式挂载到商品主档和门店商品两个对象上。

它能满足本次一期要求：

- 不分裂标签池
- 不重复建设关系表
- 不提前引入继承与聚合复杂度
- 让运营真正具备商品主档与门店商品两条链路的标签闭环能力

同时也为后续扩展到 `SKU`、规则来源、批量挂标和更细粒度对象限制保留了清晰的演进路径。
