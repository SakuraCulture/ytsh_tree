# 标签主线交接回迁设计

## 背景

当前仓库的标签主线权威基线是提交 `5048cd3`（`feat: complete product tag mainline flow`）。

用户给出的交接文档 `tag-mainline-handoff-2026-05-18.md` 约束了这次回迁的唯一目标：

- 只恢复标签主线相关内容
- 只允许迁移文档列出的 43 个文件
- 严格排除登录联调、环境配置、订单链路、脚本和临时文件
- 如果当前基线中同名文件已有新改动，必须先 diff 再决定是否合并

当前主仓库工作区已有大量未提交改动，不适合直接在原地执行回迁。因此本次采用独立 worktree 完成回迁和验证，避免污染现有 `master` 工作区。

## 目标

1. 在新的隔离 worktree 中完成标签主线 43 个文件的精准回迁。
2. 回迁内容严格以 `5048cd3` 和交接文档白名单为准。
3. 保证 SQL、后端、前端、测试的迁移顺序和验证顺序清晰可复现。
4. 不把白名单外的登录、环境、订单、脚本、临时文件混入本次结果。
5. 对白名单内与当前基线冲突的文件先 diff，再决定保留现状、直接恢复，或手工合并。

## 非目标

1. 不修复与标签主线无关的历史改动。
2. 不顺手清理当前 `master` 工作区里的其他脏改动。
3. 不修改登录、环境、部署、脚本、订单、饿了么非标签业务内容。
4. 不扩展标签主线范围之外的新能力。

## 迁移输入与权威来源

### 权威提交

- 提交：`5048cd3`
- 提交信息：`feat: complete product tag mainline flow`

### 权威文件清单

按交接文档，唯一允许迁移的文件共 43 个，按四组执行：

1. SQL
2. 后端 Java
3. 前端 API 与页面
4. 测试与测试 SQL
5. 可一并保留的上下文文档

### 白名单外明确排除项

- `ytsh-ui-vue3/src/views/Login/**`
- `tenantWebsiteProbe.ts`
- `yudao-server/src/main/resources/application*.yaml`
- `ytsh-ui-vue3/.env*`
- `yudao-module-ele/**` 中非标签内容
- `script/**`
- `.flattened-pom.xml`
- `/.tag-branch-tscheck.txt`
- 任意本地日志、导出文件、缓存、密钥或账号配置

## 总体方案

采用“新建干净 worktree + 从 `5048cd3` 按白名单精准恢复”的方案。

### 方案选择理由

相比整 commit 回放后再删无关文件，白名单精准恢复的优势是：

- 与交接文档边界完全一致
- 不会误带登录、环境、订单、脚本等白名单外改动
- 验证范围更小，问题定位更直接
- 适合当前主仓库有大量未提交改动的状态

## 执行流程

### 第 1 步：创建隔离 worktree

- 在当前仓库 `.worktrees/` 下创建新的干净 worktree
- 后续所有回迁、diff、验证都只在该 worktree 内进行
- 不在当前 `master` 工作区直接修改代码

### 第 2 步：按顺序恢复白名单文件

严格按以下顺序处理：

1. SQL
2. 后端 Java
3. 前端 API 与页面
4. 测试与测试 SQL
5. 上下文文档

恢复动作遵循两条规则：

- 如果文件当前不存在，直接从 `5048cd3` 恢复到 worktree
- 如果文件当前已存在且与 `5048cd3` 不同，先做 diff，再判断是直接采用 `5048cd3` 版本，还是保留当前内容并手工合并

### 第 3 步：冲突处理规则

若白名单内文件承载了当前基线中的其他业务改动：

1. 停止盲目覆盖
2. 列出具体文件和差异点
3. 根据差异性质决定：
   - 纯标签主线差异：采用 `5048cd3`
   - 明显属于其他业务线新增内容：优先保留现状并手工并入标签能力
   - 无法快速判断：先向用户汇报再继续

### 第 4 步：完成后验证

#### 后端最低验证

执行交接文档指定的标签相关测试：

```bash
mvn -pl yudao-module-business -Dtest=ProductSpuTagServiceImplTest,SpuTableServiceImplTest,StoreProductServiceImplTest,StoreProductTagServiceImplTest,TagValueServiceImplTest test
```

若回迁影响到 `TagObjectRelationServiceImplTest` 或其它标签核心测试，也补充执行对应定向测试。

#### 前端最低验证

至少完成以下检查：

- `ytsh-ui-vue3/src/api/business/product/index.ts`
- `ytsh-ui-vue3/src/api/business/store-product/index.ts`
- `/biz/spu-table` 标签筛选、标签列、单个挂标、批量挂标
- `/biz/store-product` 标签筛选、标签列、单个挂标、批量挂标

如果当前环境不适合完整页面联调，则至少执行标签主线相关文件的定向静态校验，并明确说明未完成的浏览器验证阻塞点。

## 风险与控制

### 风险 1：白名单文件与当前基线发生语义冲突

控制：先 diff，再合并；遇到不属于标签主线的内容不直接覆盖。

### 风险 2：把白名单外环境或登录改动误带入

控制：恢复动作只允许针对交接文档白名单；任何白名单外路径一律不动。

### 风险 3：主仓库脏工作区干扰判断

控制：全部操作放入新的隔离 worktree，避免和现有 `master` 工作区混改。

### 风险 4：验证口径不足导致“迁回去了但不可用”

控制：后端跑定向测试，前端至少完成标签主线静态核对；如能拉起页面，再验证两张标签页面的四条主链路。

## 验收标准

满足以下条件即可认为本次交接回迁完成：

1. 新 worktree 中仅包含标签主线白名单相关改动。
2. SQL、后端、前端、测试文件均已按顺序恢复。
3. 白名单内冲突文件已逐个 diff 处理，没有盲覆盖。
4. 白名单外的登录、环境、订单、脚本、临时文件没有被带入。
5. 标签相关后端定向测试通过，或明确报告失败点与原因。
6. 前端标签相关静态校验完成，若能联调则两个页面的四条主链路完成验证。
7. 最终向用户输出：变更文件范围、验证证据、剩余风险或阻塞。

## 实施边界

本设计只定义“如何把标签主线安全迁回当前仓库”。

它不是对标签功能本身的新设计，也不是对现有脏工作区的一次清理方案。后续实施必须继续遵守：

- 只改白名单文件
- 遇到白名单内冲突先 diff
- 不碰白名单外内容
- 在验证完成前不宣称作业完成
