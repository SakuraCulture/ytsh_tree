# 翱象订单页面修复与优化设计

## 背景

当前前端涉及两个翱象订单相关页面：

- `ytsh-ui-vue3/src/views/order/FailRecord.vue`
- `ytsh-ui-vue3/src/views/order/StatusLog.vue`

这两个页面目前都还是测试壳子页面，缺少正式的信息架构与交互。用户提出的目标是：

1. 先把失败记录页接口打通
2. 再把两个页面一起优化到可用状态
3. 状态变更页采用“混合型”呈现：默认简洁，详情展开后看完整信息
4. 失败记录页必须支持：查询、单条重试、批量重试

## 目标

本次改造聚焦两个页面：

- 失败记录页先解决“接口不通”，再补齐正式查询与重试操作
- 状态变更页从调试壳子升级为正式业务页面，控制主表信息密度，并提供抽屉详情

不在本次范围内的内容：

- 不改后端业务语义
- 不新增新的后端接口能力，优先复用现有接口
- 不做与本需求无关的全局样式重构

## 方案选择

采用方案 2：两页一起处理，但按“先通再优化”的顺序推进。

选择原因：

- 失败记录页存在明确阻塞问题，优先修通最能直接恢复可用性
- 状态变更页问题主要是信息组织混乱，适合在接口确认后直接按正式结构重做
- 两个页面都位于同一业务域，统一处理可以让筛选区、表格风格、反馈方式保持一致

## 页面边界与改动范围

### 失败记录页

页面职责：

- 展示失败记录分页列表
- 提供筛选查询能力
- 支持单条重试
- 支持批量重试

关键改动点：

- 修正前端 API 路径，与后端 `/ele/order/fail-record/*` 保持一致
- 将当前测试壳子替换为正式列表页
- 增加多选、批量操作、结果提示、加载态和空态

### 状态变更页

页面职责：

- 展示订单状态变更流水
- 支持按常用条件筛选
- 以简洁主表承载高频查看信息
- 通过抽屉承载完整详情

关键改动点：

- 将当前调试页替换为正式查询列表页
- 主表只保留高频字段，避免列表过于拥挤
- 通过详情抽屉展示完整快照和上下文

## 数据流设计

### 失败记录页数据流

1. 页面初始化时加载第一页数据
2. 用户调整筛选条件后触发分页查询
3. 表格支持多选，选中状态用于批量重试按钮启用/禁用
4. 单条重试成功后刷新当前页
5. 批量重试完成后刷新当前页，并提示成功/失败结果

数据流原则：

- 查询结果作为页面唯一列表数据源
- 重试操作不在前端本地改状态，统一以后端刷新结果为准
- 操作反馈明确区分“请求失败”和“业务失败”

### 状态变更页数据流

1. 页面初始化时仅拉取分页列表
2. 用户筛选后重新查询分页数据
3. 点击详情时再按 `id` 拉取单条详情
4. 详情加载成功后打开抽屉展示完整信息

数据流原则：

- 列表与详情分离，避免首屏拉取过重
- 列表只承担检索和定位职责
- 详情抽屉承担完整信息阅读职责

## 页面结构设计

### 失败记录页结构

采用“筛选区 + 操作栏 + 表格 + 分页”的标准列表页结构。

#### 筛选区

只保留高频字段，避免一屏过多条件：

- 订单号
- 业务类型
- 失败状态 / 是否可重试（如后端已有对应字段则展示）
- 时间范围（如现有接口支持）

#### 操作栏

提供：

- 批量重试按钮
- 刷新按钮

批量重试按钮默认在无选中项时禁用，减少误操作。

#### 表格

优先展示业务上最有用的信息：

- 订单号
- 业务类型
- 失败信息
- 重试次数
- 最后失败时间
- 操作

操作列至少包含“重试”。

#### 分页

沿用项目现有分页模式，保持与其他管理页一致。

### 状态变更页结构

采用“简洁主表 + 详情抽屉”的混合型结构。

#### 筛选区

保留高频检索字段，例如：

- 订单号
- 变更来源
- 状态
- 时间范围（如接口支持）

#### 主表

主表只显示高频字段：

- 订单号
- 变更前状态
- 变更后状态
- 变更来源
- 变更时间
- 操作

操作列提供“详情”。

#### 详情抽屉

详情抽屉用于承载不适合放在主表中的完整内容，例如：

- 完整状态快照
- 备注
- 原始上下文
- 关联字段补充信息

这样既保证主表清晰，也不丢失排障所需信息。

## 错误处理

### 失败记录页

需要明确区分两类问题：

1. 接口不通：通过修正 API 路径解决前后端不匹配问题
2. 接口报错：通过页面消息提示用户，并保持当前筛选状态不丢失

单条和批量重试都应有清晰反馈：

- 请求成功且业务成功：提示成功并刷新列表
- 请求成功但业务失败：提示失败原因
- 请求异常：提示重试失败，请稍后再试

### 状态变更页

- 列表查询失败时展示错误提示
- 详情加载失败时只影响抽屉，不影响当前列表浏览
- 详情按需加载，避免主页面首屏压力过大

## 测试与验收

### 失败记录页验收

必须验证：

- 查询可正常返回数据
- 单条重试可触发并返回结果
- 批量重试可基于勾选项执行
- 操作后列表能正确刷新
- 空数据、请求失败、重试失败时页面反馈正常

### 状态变更页验收

必须验证：

- 筛选与分页正常工作
- 主表展示简洁且信息足够定位问题
- 点击详情可以打开抽屉并展示完整内容
- 详情加载失败时反馈清晰

### 联调验证方式

优先使用 `playwright-cli` 做页面联调和功能完整性验证，覆盖：

- 失败记录页查询与重试主流程
- 状态变更页列表查询与详情查看主流程
- 基本空态和错误提示回归

## 实施顺序

1. 修正失败记录页 API 路径
2. 将失败记录页从测试壳子改为正式列表页
3. 打通查询、单条重试、批量重试
4. 将状态变更页从测试壳子改为正式混合型页面
5. 补齐加载态、空态、错误提示
6. 使用 `playwright-cli` 做联调验证
