# 门店库存批量导入与批量/定时拉取设计

## 背景

当前门店库存能力已经具备单次批量查询入口，能够把饿了么库存查询结果按正式库存、影子库存和治理记录分流处理。但现阶段仍缺少两类关键能力：

1. 手工批量导入门店库存。
2. 面向全量启用门店的批量任务与定时拉取能力。

如果分别为导入、手工触发任务、定时任务各写一套库存落库逻辑，后续会出现匹配规则、影子规则、治理口径和统计口径不一致的问题。因此本次设计采用统一库存流水线，把不同入口先归一为同一种标准库存行，再复用同一条正式/影子/治理分流逻辑。

## 目标

1. 支持独立的门店库存批量导入能力。
2. 支持按全量启用门店执行库存批量拉取任务。
3. 支持定时自动执行同一套库存批量任务。
4. 手工导入与 API 拉取复用统一库存写入逻辑。
5. 找不到正式门店商品时不让整批失败，而是写入库存影子并进入治理。
6. 批量作业支持门店级并发、任务级防重、进度统计、取消和失败定位。
7. 后续主档补齐或影子归正后，不需要迁移额外名单表即可继续纳入定时拉取。

## 不建设范围

1. 不自动创建正式门店商品或自动补齐商品主档。
2. 不为库存任务新增独立的长期拉取清单事实表。
3. 不把库存批量导入耦合进现有门店商品导入流程。
4. 不在第一版做多平台通用抽象，先聚焦饿了么链路。
5. 不改变现有单次库存查询页面的核心交互，只在其基础上扩展任务化能力。

## 已确认的关键决策

1. 批量导入修改的对象是门店库存。
2. 本次同时需要手工导入和基于饿了么 API 的批量/定时拉取两类入口。
3. 手工导入找不到正式门店商品时，不直接判整批失败，而是写影子并治理。
4. 定时拉取默认覆盖全量启用门店。
5. 手工导入模板要求提供完整库存字段，而不是最小字段集。
6. 并发实现需要参考现有订单和门店商品全量同步的并发模型。

## 方案对比

### 方案一：把库存字段挂进现有门店商品导入

在现有门店商品导入模板和服务里增加库存列，同时为定时拉取单独补一个后台任务。

优点：

- 表面上复用现有导入入口较多。
- 前端改动看起来较小。

缺点：

- 库存能力被耦合在商品导入里，边界不清晰。
- 手工导入和定时拉取仍会形成两套落库逻辑。
- 后续任务统计、重试、治理很难统一。

### 方案二：先做定时任务，导入后补

先把全量启用门店的库存批量/定时拉取做完，手工导入作为第二阶段能力。

优点：

- 当前实现路径最短。
- 可以优先交付自动化覆盖能力。

缺点：

- 无法一次满足本次需求。
- 第二阶段接手工导入时仍需要再梳理统一口径。

### 方案三：统一库存流水线

新增独立的库存导入与库存批量任务能力，但把 Excel 导入行和 API 拉取结果统一转换成标准库存行，再交给同一个应用服务做正式落库、影子落库和治理写入。

优点：

- 导入、手工任务、定时任务规则一致。
- 统计、失败回执、治理口径一致。
- 后续扩展其它入口时只需要增加新的数据源适配层。

缺点：

- 第一版需要补任务表、执行器、导入入口和调度入口。
- 改动面比挂靠现有导入更大。

本次选择方案三。

## 总体架构

### 核心原则

不同入口只负责提供数据，不负责决定如何落库。真正的库存业务写入只保留一条出口。

### 分层设计

1. **入口层**
   - 手工导入入口：Excel 上传并解析门店库存行。
   - 手工批量任务入口：对全量启用门店触发一次库存拉取任务。
   - 定时任务入口：按配置周期触发同一个库存拉取任务服务。

2. **标准化层**
   - 把 Excel 行和 API 返回结果统一转换成标准库存行 `EleStoreInventoryIngestRowBO`。
   - 标准库存行至少包含门店标识、SKU 标识、完整库存字段、来源类型、原始载荷和辅助定位信息。

3. **统一写入层**
   - 新增 `EleStoreInventoryIngestService` 作为统一写入边界。
   - 内部复用现有库存查询服务中的三分流语义：
     - 能匹配正式门店商品则更新正式库存。
     - 找不到正式门店商品则写库存影子。
     - 影子记录同步写入或刷新治理池。

4. **任务层**
   - 库存批量拉取使用主任务表 + 门店子任务表。
   - 门店是并发执行单元，门店内 SKU 批次串行推进。

## 批量导入设计

### 独立导入入口

库存导入不挂靠现有门店商品导入，新增独立“门店库存导入”入口，但交互风格复用现有导入弹窗与导入回执模式。

### 导入模板字段

第一版模板字段如下：

必填：

- ERP门店编码
- SKU编码
- 总库存
- 可售库存
- 占用库存
- 在途库存
- 预留库存

可选：

- 子SKU编码
- 备注

模板不要求用户手工输入 `merchantCode` 或 `platformStoreId`。系统通过 ERP 门店编码解析本地门店与平台门店映射，避免把平台映射维护成本转嫁给 Excel 使用者。

### 导入校验

导入先做两类校验：

1. **模板校验失败**
   - 必填缺失。
   - 数值格式非法。
   - 库存值为负数。
   - 库存字段之间明显矛盾。

   这类记录直接进入导入失败回执，不写业务数据。

2. **业务匹配失败**
   - 门店可识别，但找不到正式门店商品。
   - 正式库存无法安全更新。

   这类记录不算模板失败，而是进入统一库存流水线后写影子和治理。

### 导入结果回执

导入完成后至少返回：

- 正式落库成功数。
- 影子库存成功数。
- 治理记录写入数。
- 模板校验失败数。
- 逐行失败原因。

其中“写影子并治理”的记录属于业务已接收，不应混入模板失败列表。

## 批量任务与并发模型

### 任务模型

新增以下任务对象：

- `ele_store_inventory_batch_task`：库存批量任务主表。
- `ele_store_inventory_batch_task_store`：库存批量任务门店子表。

主任务保存来源、状态、门店数、SKU 数和聚合统计；子任务保存门店维度进度、当前批次、总批次和错误信息。

### 任务提交方式

采用“先建任务，再异步提交执行器”的模式：

1. 事务内创建主任务和门店子任务。
2. 事务提交后再异步启动执行器。
3. 通过分布式锁避免重复创建同范围的运行中任务。

### 并发颗粒度

并发单元选择门店，而不是单个 SKU 行。

原因：

- 饿了么库存接口天然按门店批量查询 SKU。
- 门店维度更适合做任务取消、失败重试和统计归因。
- 可以复用现有订单同步与门店商品全量同步的线程池和分批提交思路。

### 门店内执行方式

每个门店子任务内部按 SKU 集合切批，单批次调用一次库存批量接口。门店内批次串行推进，门店间并发执行。

建议执行模型：

- 子任务外层：门店并发。
- 子任务内层：SKU 批次串行。
- 每个批次处理完后刷新门店进度，再按节流频率刷新任务聚合。

### 背压与限流

批量任务执行器保留现有两层保护：

1. 线程池提交许可控制，避免本地线程池满载。
2. API 背压等待，发现接口本地排队时暂停继续提交新门店任务。

这样可以在全量启用门店定时拉取时保护线程池、限流器和上游接口。

## 定时拉取范围设计

### 门店范围

定时任务默认覆盖系统中的全量启用门店，口径与现有门店商品全量同步保持一致。

### SKU 集合来源

不新增独立拉取清单表，而是在任务执行时按门店动态汇总以下 SKU：

1. 正式门店商品中的 SKU。
2. 活跃门店商品影子记录中的 SKU。
3. 活跃库存影子或治理记录中的 SKU。

然后对 SKU 去重后分批查询。

### 这样设计的原因

- 手工导入过但尚未归正的 SKU，不会只生效一次，后续定时任务仍会继续覆盖。
- 主档补齐后，SKU 可以自然从影子来源切换到正式来源，不需要迁移名单。
- 避免维护第二份和正式/影子事实可能漂移的清单表。

## 数据对象与服务边界

### 新增 BO / VO / 配置对象

建议新增：

- `EleStoreInventoryIngestRowBO`
- `EleStoreInventoryImportReqVO`
- `EleStoreInventoryImportExcelVO`
- `EleStoreInventoryImportRespVO`
- `EleStoreInventoryBatchExecuteReqVO`
- `EleStoreInventoryBatchTaskRespVO`
- `EleStoreInventoryBatchTaskStoreRespVO`
- `EleStoreInventorySchedulerProperties`

### 新增服务边界

建议新增：

- `EleStoreInventoryIngestService`：统一库存写入服务。
- `EleStoreInventoryImportService`：导入解析与回执。
- `EleStoreInventoryBatchTaskService`：任务创建、查询、取消。
- `EleStoreInventoryBatchExecutor`：门店级并发执行器。
- `EleStoreInventorySkuScopeService`：为每个门店收集待拉取 SKU 集合。
- `EleStoreInventoryAutoSyncJob`：定时调度入口。

### 对现有服务的改造

现有单次查询服务中的正式/影子/治理分流逻辑需要下沉到可复用方法或新服务，避免：

- 单次查询一套逻辑。
- Excel 导入一套逻辑。
- 定时任务再复制一套逻辑。

第一版必须完成这一步收敛，确保未来所有入口只维护一处库存分流规则。

## 前端设计

### 新增入口

在库存相关页面新增“门店库存导入”和“库存批量任务”入口：

1. 导入入口：上传模板、下载模板、显示导入结果。
2. 任务列表：查看任务状态、来源、统计、开始结束时间。
3. 任务明细：查看门店子任务进度和失败原因。
4. 手工触发：允许人工发起一次全量启用门店库存拉取任务。

### 与现有库存查询页的关系

现有单次库存查询页保留，继续用于定位单店单批次问题。新增任务页承担批量与定时能力，不把所有能力塞进单次查询页里。

## 错误处理与一致性

### 一致性原则

1. 正式库存更新和影子/治理写入必须遵守单条标准库存行的确定性分流规则。
2. 导入和 API 拉取的同一业务事实不能因入口不同而得出不同的落库结论。
3. 主任务统计来自门店子任务聚合，不手工维护第二份统计口径。

### 失败处理原则

1. 模板校验失败：仅记录导入失败，不入业务数据。
2. 业务匹配失败：写影子和治理，不让整批失败。
3. 上游接口失败：记录到任务或门店子任务错误信息，允许后续重试。
4. 门店子任务失败：不影响其它门店继续执行。

### 取消与重试

第一版任务需要支持取消正在运行的主任务。取消后，未开始的门店子任务停止提交；已在执行中的门店子任务允许自然收尾。失败重试先保留到门店级或整任务级，不在第一版细分到单个 SKU 批次重试。

## 测试与验收策略

### 后端单测

至少覆盖：

1. Excel 导入模板校验失败回执。
2. 导入行匹配正式门店商品时更新正式库存。
3. 导入行找不到正式门店商品时写影子和治理。
4. API 拉取结果复用同一 ingest 服务时，与导入得到一致落库结论。
5. 任务创建时防重锁生效。
6. 门店子任务并发执行时，聚合统计正确。
7. SKU 集合动态汇总时，正式 SKU、门店商品影子 SKU、库存影子 SKU 能正确去重。

### 集成验证

1. 导入一批混合数据，确认正式、影子、治理和失败回执统计正确。
2. 手工触发一次全量启用门店库存批量任务，确认能创建主任务与门店子任务。
3. 模拟存在影子 SKU 的门店，确认定时任务仍会继续拉取该 SKU。
4. 同一时间重复触发同范围任务，确认分布式锁阻止重复运行。

### 前端验证

1. 模板下载、上传、导入结果展示可用。
2. 任务列表和门店子任务明细可用。
3. 失败原因、影子数量、治理数量展示清晰。
4. 手工触发任务与取消任务可用。

## 验收标准

1. 用户可以通过 Excel 批量导入门店库存。
2. 模板错误与业务影子分流结果能在回执中清晰区分。
3. 系统可以对全量启用门店发起库存批量任务。
4. 系统可以按配置周期自动执行同一套库存批量任务。
5. 导入、手工任务、定时任务复用同一条正式/影子/治理写入逻辑。
6. 批量任务采用门店级并发执行，并具备任务级防重和可观测性。
7. 定时拉取范围不依赖额外清单表，而是从正式与影子事实动态汇总 SKU 集合。

## 风险与控制

1. **风险：** 全量启用门店并发拉取可能压满线程池或接口限流。
   **控制：** 复用现有线程池、批量提交、背压等待与限流机制。

2. **风险：** 不同入口继续各自维护落库逻辑，导致口径漂移。
   **控制：** 第一版必须先收敛到统一 ingest 服务，再接入口。

3. **风险：** 手工导入模板要求完整库存字段，用户填报错误率较高。
   **控制：** 提供清晰模板、严格回执、逐行错误原因和可重复导入能力。

4. **风险：** 影子 SKU 长期累积，导致任务每次拉取范围扩大。
   **控制：** 仅纳入活跃影子和治理记录；后续再评估清理策略，但不在本次范围内。

## 结论

本次采用“统一库存流水线”方案：手工导入、手工批量任务和定时任务都先产出标准库存行，再进入统一的库存 ingest 服务，由它完成正式库存更新、影子库存写入和治理记录刷新。批量作业复用现有订单与门店商品全量同步的任务化与并发思路，按门店并发、按 SKU 批次串行推进，并通过分布式锁、线程池背压和任务聚合统计保证可控性与可观测性。
