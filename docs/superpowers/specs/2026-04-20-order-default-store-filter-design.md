# 订单页默认门店筛选设计

## 背景

当前订单页 `ytsh-ui-vue3/src/views/order/index.vue` 在首次进入时会自动查询订单数据。页面已经默认按“今日”设置日期范围，但门店筛选项初始为空，因此首次请求会查询“今日全部门店”的订单数据。

这会带来两个问题：

- 首屏打开时对后端查询压力偏大
- 用户真正高频关注的通常是某一家正在营业门店的当日订单，而不是全部门店汇总

本次目标是在不改变用户后续手动筛选能力的前提下，收敛默认查询范围，降低首屏查询压力。

## 目标

本次改动目标：

- 首次进入订单页时，默认查询第一家开业状态门店的今日订单
- 用户点击“重置”后，恢复为同样的默认条件
- 保持现有日期快捷筛选逻辑不变，仍以“今日”为默认时间范围
- 不影响用户手动切换到其他门店或清空门店后的主动查询行为

不在本次范围内的内容：

- 不修改后端接口或查询语义
- 不新增全局配置项来指定默认门店
- 不调整订单列表的展示结构和其他筛选项行为

## 方案选择

本次采用方案 1：仅在首屏初始化与重置时自动选中第一家开业门店。

对比备选方案：

1. 仅在首屏初始化与重置时回填默认门店
2. 在 `getList()` 内对空门店做统一兜底
3. 增加单独的初始化标记，区分首屏与后续手动清空场景

选择方案 1 的原因：

- 改动最小，逻辑清晰，直接覆盖“首屏”和“重置”两个需要默认值的入口
- 不会干扰用户后续主动清空门店并执行查询的行为
- 比在 `getList()` 中统一兜底更安全，避免每次查询都强制带回默认门店

## 业务规则

### 默认门店选择规则

门店列表加载成功后：

- 按现有逻辑先对门店按 `storeStatus` 排序，开业状态在前
- 从排序后的门店列表中找到第一条 `storeStatus === 1` 的记录
- 将其 `platformStoreId` 作为默认 `storeId`

这意味着页面默认只会命中第一家开业状态门店，而不是所有门店。

### 无开业门店时的处理

如果门店列表中不存在任何开业状态门店：

- 保持 `queryParams.storeId` 为空
- 不额外退化为“自动查询全部门店”
- 页面继续沿用现有时间默认值，但避免通过默认门店逻辑触发更大范围查询

该规则的核心目的是避免因为默认值缺失而重新回到首屏大查询。

### 重置行为

用户点击“重置”时：

- 恢复 `dateType = 'today'`
- 恢复其他筛选项为空
- 重新按当前已加载的门店列表，回填第一家开业门店到 `storeId`
- 再执行查询

这样“首屏”和“重置”行为保持一致，用户心智更稳定。

## 数据流设计

### 首屏初始化流程

1. 页面创建时先加载门店列表
2. 门店列表返回后完成排序
3. 从排序结果中计算默认开业门店
4. 将默认门店写入 `queryParams.storeId`
5. 再执行 `getList()`，由现有日期逻辑同步出“今日”的时间范围
6. 请求参数中同时携带 `startTime`、`endTime` 和 `platformStoreId`

### 重置流程

1. 用户点击“重置”
2. 页面将筛选项恢复为默认值
3. 根据当前 `storeList` 重新计算默认开业门店
4. 写回 `queryParams.storeId`
5. 执行查询

### 用户主动操作流程

用户后续手动修改门店、清空门店或切换其他筛选项时：

- 不再额外插入默认门店逻辑
- 保持当前页面行为，以用户操作结果为准

## 改动范围

主要改动集中在：

- `ytsh-ui-vue3/src/views/order/index.vue`

涉及的关键位置：

- 门店列表加载完成后的初始化逻辑
- 查询参数默认值构造逻辑
- 重置查询逻辑

本次不需要修改 API 文件，也不需要改动后端接口。

## 实现边界

### 需要保持不变的行为

- 日期默认值仍为“今日”
- 搜索按钮、标签页切换、分页逻辑保持现有行为
- 用户手动选择门店后，查询参数以用户选择为准
- 用户手动清空门店后，不自动再次填回默认门店，除非点击“重置”

### 需要避免的问题

- 不要把默认门店兜底写进所有查询路径，否则会破坏“手动清空门店”的语义
- 不要在没有开业门店时自动退化为全门店默认查询
- 不要因为初始化顺序问题导致在门店列表未加载完成前先发起全量订单查询

## 验收标准

必须满足以下验收结果：

1. 首次进入订单页时，请求参数默认带上第一家开业门店的 `platformStoreId`
2. 首次进入订单页时，日期范围仍为当天
3. 点击“重置”后，请求参数恢复为“今日 + 第一家开业门店”
4. 用户手动切换门店后查询正常
5. 用户手动清空门店后，页面不会偷偷恢复默认门店，除非再次点击“重置”
6. 当不存在开业门店时，不因为默认逻辑触发全门店大查询

## 测试建议

建议至少验证以下场景：

- 门店列表同时存在开业和闭店门店时，默认命中第一家开业门店
- 门店列表全部为非开业门店时，页面不会因为默认逻辑扩大查询范围
- 点击重置后，默认门店与日期恢复正确
- 用户手动清空门店并搜索时，行为符合手动输入语义
- 切换标签页和分页后，不影响已选门店条件
